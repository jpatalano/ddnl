Dataset API

Build and populate datasets that power dashboards, reports, and analytics in DDNL. All operations are scoped to your instance — your API key determines which tenant's data you access.

Base URL https://fcc-bi.staturedev.com

Authentication

Every request must include your API key in the X-Api-Key header. The key encodes your tenant identity — no additional tenant ID is needed.

X-Api-Key: ik_your_api_key_here
Key scoping: Each API key is bound to a single client (tenant). All reads and writes through a key are automatically isolated to that client's data.

Field Types

Used when defining schema fields in POST /admin/datasets and POST /admin/datasets/:name/publish.

TypeDescriptionExample values
keywordExact-match string. Use for IDs, status codes, categories, names."Active", "INV-001"
textFull-text string. Use for free-form notes or descriptions."Delivered on time"
numberNumeric value. Integers and decimals. Used for metrics and aggregations.1250.00, 42
dateISO 8601 date/datetime. Used as the primary time axis for charts."2026-04-01", "2026-04-01T08:00:00Z"
booleanTrue/false flag.true, false

Filter Syntax

Filters are used in the soft-delete endpoint. Each filter is an object with three fields:

{
  "segmentName": "Status",   // field name (case-sensitive, matches your schema)
  "operator":    "eq",       // see operators below
  "value":       "Inactive"  // scalar or array depending on operator
}
OperatorMeaningValue type
eqEqualsscalar
neqNot equalsscalar
inIn listarray
not_inNot in listarray
gtGreater thanscalar
gteGreater than or equalscalar
ltLess thanscalar
lteLess than or equalscalar

Datasets

POST /api/ingest/admin/datasets Create a dataset and its initial schema

Creates the dataset definition, publishes schema version 1, and provisions the underlying search index. This is the first step before you can push any data.

Request body

FieldTypeDescription
name requiredstringImmutable key. Lowercase, underscores only. Used in all subsequent API calls. Cannot be changed after creation.
label requiredstringDisplay name. Shown in the UI. Can be updated later.
descriptionstringOptional longer description shown in Designer.
fields requiredarrayInitial schema. At least one field required. See field object below.
dataset_typestringclient (default) or provided. Use client for data you own and push. provided is reserved for platform-managed datasets.

Field object

FieldTypeDescription
name requiredstringField key. Must be unique within the schema. Used in all filter and grouping references.
type requiredstringOne of: keyword, text, number, date, boolean. See Field Types above.
primary_keybooleanMark true on one or more fields to enable upsert mode. Records with matching PK values are updated in place rather than duplicated.
date_fieldbooleanMark true on the primary date field. Used as the time axis for all charts and date-range filters. Only one field should carry this flag.

Example

POST /api/ingest/admin/datasets
X-Api-Key: ik_your_key

{
  "name":  "work_orders",
  "label": "Work Orders",
  "description": "Equipment work order records",
  "fields": [
    { "name": "WorkOrderId",  "type": "keyword", "primary_key": true },
    { "name": "YardId",       "type": "keyword" },
    { "name": "Status",       "type": "keyword" },
    { "name": "EquipmentId",  "type": "keyword" },
    { "name": "OpenDate",     "type": "date",    "date_field": true },
    { "name": "CloseDate",    "type": "date" },
    { "name": "LaborHours",   "type": "number" },
    { "name": "PartsCost",    "type": "number" },
    { "name": "Notes",        "type": "text" }
  ]
}

Response

{
  "success": true,
  "dataset": {
    "id": 7,
    "name": "work_orders",
    "label": "Work Orders",
    "current_version": 1,
    "esAlias": "fcc__work_orders",
    "esIndex": "fcc__work_orders_v1"
  }
}

POST /api/ingest/admin/datasets/:name/publish Publish a new schema version

Adds fields or changes types by publishing a new schema version. The API runs a compatibility check — additive changes (new fields) are non-breaking. Removing or retyping fields is flagged as breaking and blocked unless force: true is passed.

Breaking changes: Removing a field or changing its type from number to keyword (or vice versa) will be rejected without "force": true. Existing data is never modified by a schema publish.

Request body

FieldTypeDescription
fields requiredarrayComplete new field list (same format as create). Must include all existing fields plus any additions.
forcebooleanSet true to override breaking-change rejection. Use with caution.

Example — add a field

POST /api/ingest/admin/datasets/work_orders/publish
X-Api-Key: ik_your_key

{
  "fields": [
    { "name": "WorkOrderId", "type": "keyword", "primary_key": true },
    { "name": "YardId",      "type": "keyword" },
    { "name": "Status",      "type": "keyword" },
    { "name": "EquipmentId", "type": "keyword" },
    { "name": "OpenDate",    "type": "date", "date_field": true },
    { "name": "CloseDate",   "type": "date" },
    { "name": "LaborHours",  "type": "number" },
    { "name": "PartsCost",   "type": "number" },
    { "name": "Notes",       "type": "text" },
    { "name": "TechnicianId","type": "keyword" }
  ]
}

Response

{
  "success": true,
  "version": 2,
  "compat":  "ok",
  "notes":   ["Added field: TechnicianId (keyword)"]
}

Data Ingest

Upsert vs. append: If your dataset has one or more primary_key fields defined, all bulk and single pushes run in upsert mode — a deterministic ID is derived from the PK values, so re-sending the same record updates it in place. Without a PK, records are always appended.
Preferred format: JSON. Use /bulk or /single for all integrations. CSV upload exists as a fallback but is not recommended — it requires file handling, offers no upsert guarantees, and is harder to automate reliably at scale.
POST /api/ingest/:dataset/bulk Push multiple records — upsert or replace

The primary ingest endpoint. Designed for high volume — send thousands of records per call. Uses upsert mode when the dataset has a primary key defined.

Request body

FieldTypeDescription
docs requiredarrayArray of record objects. Field names must match your schema exactly.
replacebooleanDefault false. Set true to wipe the entire index and replace with this payload. Use for full daily loads.

Example

POST /api/ingest/work_orders/bulk
X-Api-Key: ik_your_key
Content-Type: application/json

{
  "docs": [
    {
      "WorkOrderId":  "WO-2026-001",
      "YardId":       "YARD-ATL",
      "Status":       "Closed",
      "EquipmentId":  "EQ-1042",
      "OpenDate":     "2026-03-10",
      "CloseDate":    "2026-03-14",
      "LaborHours":   12.5,
      "PartsCost":    340.00
    },
    {
      "WorkOrderId":  "WO-2026-002",
      "YardId":       "YARD-MIA",
      "Status":       "Open",
      "EquipmentId":  "EQ-0887",
      "OpenDate":     "2026-04-01",
      "LaborHours":   0,
      "PartsCost":    0
    }
  ]
}

Response

{
  "success":    true,
  "indexed":    2,
  "failed":     0,
  "errors":     [],
  "durationMs": 84,
  "mode":       "upsert"
}
modeMeaning
upsertPK-based — re-sending the same WorkOrderId updates in place
replaceFull index swap — all prior records removed
appendNo PK defined — every call adds new records

Records

GET /api/ingest/:dataset/records/:id Fetch a single record by its internal ID new

Fetches a single document by its deterministic internal ID (the SHA-256 derived from your primary key values). Soft-deleted records return 404.

Getting the ID: The ID is returned as id in the single-upsert response. For bulk upserts, derive it the same way the API does: SHA-256( clientId + "|" + datasetName + "|" + pkValue1 + "|" + pkValue2 ) — hex encoded.

Example

GET /api/ingest/work_orders/records/a3f9c12e4b...
X-Api-Key: ik_your_key

Response

{
  "success": true,
  "id": "a3f9c12e4b...",
  "doc": {
    "WorkOrderId": "WO-2026-003",
    "YardId":      "YARD-SEA",
    "Status":      "Open",
    "OpenDate":    "2026-04-29",
    "_status":     "active",
    "__ingested_at": "2026-04-29T07:15:00Z"
  }
}
POST /api/ingest/:dataset/delete Soft-delete records by filter or ID list new

Marks matching records as deleted by setting _status = "deleted". Records are never physically removed — they are excluded from all queries, charts, and reports automatically.

This is irreversible via the API. Deleted records can be restored directly in the database if needed, but there is no API-level undelete today.

Request body

FieldTypeDescription
filtersarrayOne or more filter objects (see Filter Syntax above). At least one of filters or ids is required.
idsarrayArray of internal record IDs (SHA-256 hex strings) to delete directly.

Example — delete by filter

POST /api/ingest/work_orders/delete
X-Api-Key: ik_your_key
Content-Type: application/json

{
  "filters": [
    { "segmentName": "Status",  "operator": "eq",  "value": "Cancelled" },
    { "segmentName": "OpenDate","operator": "lt",  "value": "2024-01-01" }
  ]
}

Example — delete by IDs

{
  "ids": ["a3f9c12e...", "b7d4a091..."]
}

Response

{
  "success": true,
  "deleted": 14,
  "noops":   0
}

Relations

Relations declare how datasets are connected. Once declared, DDNL automatically enriches records at upsert time — no extra work required from the sender.

How pull_fields works: When a Job record is pushed with CustomerId: "CUST-042", DDNL looks up the customers dataset, finds the matching record, and stamps the declared pull fields (e.g. CustomerName, Region) directly onto the Job before indexing. The sender never needs to include those fields — they're resolved automatically.
Source wins: If the incoming record already includes a pull field, the sender's value is kept as-is. The lookup only fills fields that are absent or null.

Relation types

TypeMeaningpull_fields
belongs_toMany-to-one. Source has a FK pointing at one record in the target. e.g. Job → Customer, Job → Yard.Supported — fields are stamped at upsert time
has_manyOne-to-many. Source is the parent; target records FK back to it. e.g. Invoice → Lines.Not applicable — query-time join (future)

Returns all relations declared on this dataset as the source.

POST /api/ingest/admin/datasets/:name/relations Declare a relation new

Declares a relation from this dataset to another. Upsert with the same sourceField + targetDataset combination will update the existing relation in place.

Request body

FieldTypeDescription
sourceField requiredstringThe FK field on this dataset's records. e.g. "CustomerId"
targetDataset requiredstringThe dataset name (key) to relate to. e.g. "customers"
targetField requiredstringThe field on the target dataset to match against. Typically its primary key. e.g. "CustomerId"
relationTypestringbelongs_to (default) or has_many
pullFieldsarrayField names from the target dataset to auto-stamp onto source records at upsert time. Only for belongs_to. e.g. ["CustomerName", "Region"]
labelstringOptional display name for this relation. e.g. "Customer"

Example — Job belongs to Customer, pull name + region

POST /api/ingest/admin/datasets/jobs/relations
X-Api-Key: ik_your_key
Content-Type: application/json

{
  "sourceField":   "CustomerId",
  "targetDataset": "customers",
  "targetField":   "CustomerId",
  "relationType":  "belongs_to",
  "pullFields":    ["CustomerName", "Region"],
  "label":         "Customer"
}

After this is declared, every Job upsert that includes a CustomerId will automatically have CustomerName and Region stamped on the record — no changes needed in the sending system.

Example — Invoice has many Lines

{
  "sourceField":   "InvoiceId",
  "targetDataset": "invoice_lines",
  "targetField":   "InvoiceId",
  "relationType":  "has_many",
  "label":         "Lines"
}

Response

{
  "success":  true,
  "relation": {
    "id":             1,
    "source_field":   "CustomerId",
    "target_dataset": "customers",
    "target_field":   "CustomerId",
    "relation_type":  "belongs_to",
    "pull_fields":    ["CustomerName", "Region"],
    "label":          "Customer"
  }
}
DELETE /api/ingest/admin/datasets/:name/relations/:id Remove a declared relation

Removes the relation. Existing records that were already enriched are not affected — only future upserts will no longer receive the pull fields.

Example

DELETE /api/ingest/admin/datasets/jobs/relations/1
X-Api-Key: ik_your_key

Response

{ "success": true }

Generates a new API key for the authenticated tenant. The raw key is returned once only — store it securely. Subsequent calls to list keys return only the last 8 characters for identification.

Request body
